Analysis

Open street map is a 2dimensional visual of the world, it contains just x and y axis no z axis. There is no way to represent heigh explicitly.

nodes are points that are a set of coordinates.

nodes can be a tree or lampost etc to identify nodes OpenStreetMap data uses tags. Tags tell the computer what a point or a node on a map represents. Tags consists of a key and value. For example for a node that represents a tree it will contain a key=natural and value=tree, so the tag for a tree node is natural=tree (key=value). Tags are unsually lowercase and use underscores instead of spaces. You can have more than one tag associated with one node.

Nodes are great for points but for lines such as roads and rivers they are not as useful, instead ways are used. Ways are lines that are a collection of nodes, the connection represents a collection of nodes. The way contains the order of which the nodes appear. The order is significant as it shows the direction, the order tells the computer what shape the line is.A way is an ordered list of nodes. Ways have the same tagging system as nodes they have a key value pair. If the way was a river for example then the key is waterway and value is river, waterway= river.

Polygons also use ways but they are closed. So all nodes are joined.

Nodes have id numbers associated with them that are independent of the ordering we give them in a way. Way also have ids which represents the collection of nodes in the way.

For ways that cross each other the computer can route them together to form one node at the intersection so that they share a common node.

Layers help uncover height due to not having z axis. Layers work from ‘ground level’ or ‘level 0’ . layers occur when two ways cross over each other(intersect) but cannot be routed, so instead of two roads meeting at a junction where they ca share a node, two roads may cross over each other at different heights so will not share a node therefore cant be routed and needs layers instead. For example a river which is on way with a bridge overlapping, second way, would need layers.

Ways can be split using layers with new tags and adding new nodes where the split may occur.

Relations can contain nodes, ways and other relations. Relations describe the relationship between objects, they contain other objects the same as ways containing other nodes. They can be identified by a type and then the relation can be tagged. Roles are used to identify what role which object has for example if the type is a multi-polygon the the objects may have an inner polygon or outer polygon role.

Buildings can have addresses, these are often tagged at the entrances so that in the case where the building can be entered from multiple different streets then each entrance will be a node that is tagged.

Address interpolation enables the computer to fill out addresses for you, so it can recognise from a few address what the address of buildings in between would be. This is done through creating a way between two known address.

Addresses are attached to street through relations. The type would be associatedStreet. This is to prevent the computer from just randomly picking the nearest street to that building as for example a building may be entered from a street in front but there is also a different street close behind the building, the computer can get confused, relations are used to stop this confusion.

The platform integrates spatial data from OpenStreetMap (OSM) and other open data sources to provide designers with interactive tools for viewing environment conditions, infrastructure, and land use. The section outlines the methodological approach used to source, process and visualise spatial data within a custom-built web platform, as well as the the logic behind technical decisions.

OpenStreetMap (OSM) is a community-driven, open source geographic database that provides detailed spatial data world wide. OSM represents real world features using three primary data structures:

Nodes
Individual points defined by geographical coordinates. These are used to represent discrete features such as trees, lampposts or bus stops.

Ways
Ordered lists of nodes that represent linear or polygonal features such as roads, rivers, or building outlines. If a way forms a closed loop, it can define an area.

Relations
Groups of nodes and ways that define more complex relationships between features. For example, a relation might describe a building with multiple parts or a bus route composed of several road segments.

Each of these elements is tagged with key-value pairs (e.g. building=yes, landuse=grass, natural=tree) that describe their function or classification. These tags form the foundation for filtering and visualising relevant data within the platform. Understanding this structure was essential for filtering specific types of features such as green spaces, transportation networks, or building footprints and converting them into formats suitable for browser based visualisation.

Overpass API can be used to allow custom queries on OSM data using tag based filters.

While OSM provides comprehensive coverage, data completeness and accuracy can vary by region. Therefore, visualised datasets may be supplemented or validated using official sources where available. Additionally, OSM does not include elevation (z-axis) by default, which limits direct representation of topography or building height unless derived from other datasets.
